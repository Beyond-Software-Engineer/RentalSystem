## 一、项目概述

### 1.1 项目简介

本项目为租房系统H5移动端应用，采用前后端分离架构，前端基于Vue3构建单页面应用(SPA)，后端基于Spring Boot4构建RESTful API服务。

### 1.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | Vue3 + Vite |
| 前端路由 | Vue Router4 |
| 状态管理 | Pinia |
| UI组件库 | Vant4 |
| HTTP客户端 | Axios |
| 后端框架 | Spring Boot4 |
| ORM框架 | MyBatis Plus |
| 数据库 | MySQL8.0 |
| 安全框架 | Spring Security + JWT |
| 构建工具 | Maven |

## 三、代码规范

### 3.1 通用规范

1. **缩进**：使用4个空格进行缩进，禁止使用Tab
2. **行宽**：单行代码不超过120个字符
3. **命名**：见下文详细规范
4. **注释**：
   - 类和方法必须添加Javadoc注释
   - 关键业务逻辑必须添加代码注释
   - 注释应说明"为什么"而非"是什么"
5. **日志**：
   - 使用@Slf4j注解
   - 生产环境禁止使用System.out
   - 日志级别：DEBUG < INFO < WARN < ERROR

### 3.2 后端Java规范

#### 3.2.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | UpperCamelCase | UserController |
| 方法名 | lowerCamelCase | getUserById |
| 变量名 | lowerCamelCase | userName |
| 常量 | UPPER_SNAKE_CASE | MAX_PAGE_SIZE |
| 包名 | 全小写 | com.rental.controller |
| 数据库表名 | 小写下划线 | sys_user |
| 字段名 | 小写下划线 | user_name |

#### 3.2.2 类命名规范

| 类型 | 后缀 | 示例 |
|------|------|------|
| 控制器 | Controller | UserController |
| 服务接口 | Service | UserService |
| 服务实现 | ServiceImpl | UserServiceImpl |
| 数据访问接口 | Mapper | UserMapper |
| 实体类 | Entity | UserEntity |
| DTO | DTO | UserDTO |
| VO | VO | UserVO |
| 响应结果 | Response | CommonResponse |

#### 3.2.3 方法命名规范

| 操作 | 命名规则 | 示例 |
|------|----------|------|
| 新增 | save/add + 实体名 | saveUser |
| 修改 | update + 实体名 | updateUser |
| 删除 | delete/remove + 实体名 | deleteUser |
| 查询单个 | get + 实体名 + By + 条件 | getUserById |
| 查询列表 | list + 实体名 + (条件) | listUser |
| 分页查询 | page + 实体名 | pageUser |
| 统计 | count + 实体名 | countUser |
| 是否存在 | exists + 实体名 | existsUser |

#### 3.2.4 代码示例

```java
/**
 * 用户控制器
 *
 * @author system
 * @since 2026-05-23
 */
@RestController
@RequestMapping("/api/user")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    /**
     * 根据ID查询用户
     *
     * @param id 用户ID
     * @return 用户信息
     */
    @GetMapping("/{id}")
    public CommonResponse<UserVO> getUserById(@PathVariable Long id) {
        return CommonResponse.success(userService.getUserById(id));
    }

    /**
     * 分页查询用户列表
     *
     * @param pageNum  页码
     * @param pageSize 每页数量
     * @param query    查询条件
     * @return 用户分页列表
     */
    @GetMapping("/page")
    public CommonResponse<PageResult<UserVO>> pageUser(
            @RequestParam(defaultValue = "1") Integer pageNum,
            @RequestParam(defaultValue = "10") Integer pageSize,
            UserQueryDTO query) {
        return CommonResponse.success(userService.pageUser(pageNum, pageSize, query));
    }
}
```

#### 3.2.5 分层职责

- **Controller层**：参数校验(@Valid)、调用Service、异常处理
- **Service层**：业务逻辑处理、事务管理(@Transactional)
- **Mapper层**：数据访问，由MyBatis Plus自动实现

## 四、Git协作规范

### 4.1 分支管理策略

采用Git Flow分支模型：

| 分支类型 | 命名规范 | 说明 |
|----------|----------|------|
| 主分支 | main | 正式环境代码，永远保持可发布状态 |
| 开发分支 | develop | 开发主分支，集成所有功能 |
| 功能分支 | feature/功能名称 | 开发新功能 |
| 修复分支 | fix/问题描述 | 修复生产问题 |
| 发布分支 | release/版本号 | 发布准备 |

### 4.2 分支操作流程

```bash
# 1. 从develop创建功能分支
git checkout -b feature/user-login develop

# 2. 开发完成后，提交代码
git add .
git commit -m "feat: 添加用户登录功能"

# 3. 推送到远程
git push -u origin feature/user-login

# 4. 合并到develop
git checkout develop
git merge --no-ff feature/user-login
git branch -d feature/user-login
```

### 4.3 提交信息规范

采用Angular提交规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

#### Type类型：

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复bug |
| docs | 文档变更 |
| style | 代码格式（不影响功能） |
| refactor | 重构 |
| perf | 性能优化 |
| test | 测试相关 |
| build | 构建相关 |
| ci | CI/CD相关 |
| chore | 其他变更 |

#### 示例：

```
feat(auth): 添加手机验证码登录功能

- 实现短信验证码发送接口
- 添加验证码60秒倒计时限制
- 支持手机号合法性校验

Closes #123
```

### 4.4 代码审查流程

1. **Pull Request创建**：从功能分支合并到develop必须创建PR
2. **审查要求**：
   - 代码风格符合规范
   - 单元测试覆盖率达标
   - 无明显性能问题
   - 业务逻辑正确
3. **合并条件**：至少1人review通过方可合并

## 六、数据库设计规范

### 6.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 数据库 | 小写下划线 | rental_system |
| 表名 | 小写下划线_模块 | sys_user, biz_house |
| 字段名 | 小写下划线 | user_name |
| 索引名 | idx_表名_字段名 | idx_sys_user_phone |
| 外键名 | fk_表名_关联表名 | fk_sys_user_role |
| 唯一约束 | uk_表名_字段名 | uk_sys_user_phone |

### 6.2 表设计规范

1. 必须包含的字段：
   - id：主键，BIGINT，自增
   - create_time：创建时间，DATETIME
   - update_time：更新时间，DATETIME
   - create_by：创建人，BIGINT
   - update_by：更新人，BIGINT
   - deleted：逻辑删除标记，TINYINT

2. 字段规范：
   - 字符串类型使用varchar，长度合理设置
   - 金额使用DECIMAL(10,2)
   - 时间使用DATETIME
   - 状态字段使用TINYINT

### 6.3 索引规范

1. 根据查询需求创建索引
2. 避免在频繁更新的字段上建索引
3. 复合索引遵循最左前缀原则
4. 定期分析慢查询日志优化索引

## 七、安全规范

### 7.1 认证与授权

1. 使用JWT实现无状态认证
2. Token有效期：access_token(2小时)，refresh_token(7天)
3. 敏感操作需要重新验证

### 7.2 接口安全

1. 所有接口需要认证（公开接口除外）
2. 参数必须校验，防止SQL注入和XSS攻击
3. 敏感数据脱敏返回

### 7.3 密码安全

1. 密码加密存储（BCrypt）
2. 密码强度校验

## 八、跨端协作标准

### 8.1 接口约定

1. 前后端通过JSON格式交互
2. 日期格式：yyyy-MM-dd HH:mm:ss
3. 枚举值使用整型数字
4. 列表查询必须支持分页

### 8.2 版本控制

1. API版本通过URL路径控制：/api/v1/user
2. 重大变更需要版本升级
3. 旧版本需保留至少3个月

### 8.3 文档要求

1. 使用Swagger/OpenAPI生成接口文档
2. 接口文档需包含：请求参数、响应格式、错误码
3. 文档需保持与代码同步更新
